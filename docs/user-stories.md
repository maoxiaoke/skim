# Skim — Skills 管理器 · 用户故事与决策文档

> 阶段：Scrum Phase 2 产出 ｜ 日期：2026-06-12 ｜ 状态：待评审
> 下一阶段：PRD → 多组原型参考 → 技术方案

## 1. 背景与问题

Claude Code / Codex 用户的本地 skills 随时间膨胀（本机实测：`~/.claude/skills` 32 个、`~/.agents/skills` 31 个、`~/.codex/skills` 4 个），其中混杂测试垃圾、重复描述、游离文件。官方均未提供管理界面，用户只能手动 `rm`（危险）或放任不管（污染上下文、触发混乱）。

**Skim 的定位**：跨 agent 的本地技能生命周期管理器 —— 盘点、禁用、归档、删除，一处搞定。

## 2. 目标用户与交付形态

| 决策项 | 结论 | 关键理由 |
|---|---|---|
| 目标用户 | 自己 + 公开发布，**覆盖普通用户**（非终端重度人群） | 生态位空白；CLI 门槛被判定为过高 |
| 交付形态 | **Tauri 桌面应用**（dmg / Homebrew cask 分发） | 包体 5-10MB vs Electron 100MB+；95% 开发时间在熟悉的前端栈 |
| 架构约束 | **领域逻辑纯函数化**（frontmatter 解析、状态判断、操作计划生成与文件 IO 分离） | 未来低成本补 CLI；可测试性 |
| 签名公证 | 初期不买证书，Homebrew cask / xattr 过渡；有声量后再买（$99/年） | 所有桌面形态共同成本，可推迟 |

## 3. 管理范围

| Agent | 用户级 | 项目级 | 不管 |
|---|---|---|---|
| Claude Code | `~/.claude/skills/` | `<project>/.claude/skills/` | plugin 附带 skills（已有 `enabledPlugins` 官方开关） |
| Codex | `~/.codex/skills/`、`~/.agents/skills/` | `<project>/.agents/skills/`（及 git 上级目录） | bundled system skills（`.system/`，仅展示不删） |

**项目发现**：自动从两家 agent 的会话记录（`~/.claude/projects/` 目录名编码、`~/.codex/session_index.jsonl`）扫出用过的项目并过滤出含技能目录的 + 支持手动添加文件夹。

## 4. 状态模型（核心设计）

经查证，**两家 agent 均有原生禁用机制**，disable 一律走原生配置、不动文件：

| 操作 | 机制 | 文件 |
|---|---|---|
| **enable / disable** | Claude：`settings.json` 的 `skillOverrides`（四档：`on` / `name-only` / `user-invocable-only` / `off`）；项目级写 `.claude/settings.local.json`（git 不跟踪，天然无冲突）。Codex：全局 `~/.codex/config.toml` 的 `[[skills.config]] enabled = false`（项目级配置有上游 bug，统一写全局+绝对路径） | 不动 |
| **archive** | 移到 `~/.skim/archive/<命名空间>/<skill>/` + manifest 记录来源路径、agent、时间；可一键恢复原位；可跨机备份 | 移走留底 |
| **delete** | 移入系统废纸篓（绝不 `rm -rf`） | 系统级后悔药 |

Claude 的四档中间态在 UI 暴露：默认交互简化为开关，高级模式可选 `name-only` / `user-invocable-only`（上下文占用优化是重度用户真痛点）。

## 5. 用户故事（MVP）

### US-1 统一盘点
**作为**多 agent 用户，**我想要**打开应用即看到所有 agent、所有层级技能的统一列表（agent 来源、作用域、名称、描述、状态、体积），**以便**对我机器上的技能资产有全局认知。
- ✅ 零配置首屏即有数据（自动扫描用户级目录 + 自动发现的项目）
- ✅ 可按 agent / 作用域 / 状态 筛选，按名称搜索
- ✅ 同名/重复 description 的技能有视觉提示（如本机三个 "A test skill"）

### US-2 查看详情
**作为**用户，**我想要**点开任一技能查看渲染后的 SKILL.md、frontmatter 元信息、文件清单和体积，**以便**在禁用/删除前知道它是什么。
- ✅ SKILL.md 按 Markdown 渲染；frontmatter 表格化展示
- ✅ 显示目录文件树与总体积
- ✅ 提供"在 Finder / 文件管理器中打开"入口

### US-3 禁用与启用
**作为**用户，**我想要**一键开关技能且不动任何文件，**以便**安全地做减法、随时反悔。
- ✅ 开关写入对应 agent 的原生配置（见 §4），立即生效（Codex 提示需重启会话）
- ✅ Claude 技能高级模式可选四档；列表中清晰标识当前档位
- ✅ 已被用户在配置中手工禁用的技能，Skim 能正确读出状态（双向同步，不覆盖未知配置）

### US-4 归档
**作为**用户，**我想要**把长期不用的技能移到中央归档区留底，**以便**目录清爽又不怕哪天要找回。
- ✅ 归档后原目录消失，agent 不再加载；归档区可浏览
- ✅ 一键恢复到原路径；原路径已存在同名时给出冲突处理选项
- ✅ manifest 记录：源路径、agent、归档时间；不同来源同名技能不冲突

### US-5 删除
**作为**用户，**我想要**删除明确无用的技能且有后悔药，**以便**放心清理。
- ✅ 删除 = 移入系统废纸篓，二次确认对话框中展示技能名与体积
- ✅ 支持从详情页和列表两处触发

### US-6 批量操作
**作为**积攒了几十个技能的用户，**我想要**多选后批量禁用/归档/删除，**以便**一次大扫除不用逐个点。
- ✅ 列表支持多选（含按筛选结果全选）
- ✅ 批量操作前展示操作摘要清单并确认
- ✅ 批量结果有逐项成功/失败反馈

### US-7 项目管理
**作为**多项目用户，**我想要**应用自动列出我用过 agent 的项目并允许手动补充，**以便**项目级技能也纳入管理。
- ✅ 自动发现的项目必须**校验目录真实存在**，不存在的项目不展示（含手动添加后被删除/移动的项目）
- ✅ 自动发现的项目按"含技能数"排序展示
- ✅ 支持手动添加 / 移除项目文件夹（移除仅影响 Skim 视图，不动磁盘）
- ✅ 提供**手动刷新**入口（重新执行项目发现 + 目录存在性校验 + 技能扫描）
- ✅ 支持**定时自动刷新**（默认开启，间隔可配置），外部变化（新装技能、删除项目）无需重启应用即可反映

## 6. v2 Backlog（明确不做进 MVP）

- 健康检查 / lint（缺 frontmatter、重复 description、游离文件如 `rams.md`、体积异常）
- 跨 agent 复制 / 同步技能；用户级 ↔ 项目级移动
- 使用频率统计（解析 session transcript，标记"从未触发"的技能）
- CLI 形态（架构已预留：领域逻辑纯函数化）
- 技能市场 / 安装来源集成

| # | 假设 / 风险 | 验证方式 | 阶段 |
|---|---|---|---|
